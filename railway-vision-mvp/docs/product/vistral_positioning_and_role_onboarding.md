# Vistral 产品定位与角色上手手册

- Owner: Product + Engineering
- Status: In Use
- Last Updated: 2026-03-06
- Scope: 对外产品定位、角色价值表达、首屏与首条路径设计依据
- Non-goals: 不替代架构文档、接口文档、合同边界文档

## 1. 一句话定位

Vistral 是一个面向铁路、政企内网和受监管场景的 **模型主权 + 数据主权 + 供应商受控协作 + 边缘安全交付** 平台。

平台重点解决以下问题：

- 数据不出域，训练与交付过程始终处于平台控制面下。
- 供应商只提供算法能力，不掌握生产发布权、签名权、收费权。
- 成果模型由平台审批、签名、加密、授权并下发到指定设备。
- 边缘设备在离线、弱网、长期运行环境下仍能完成拉取、验签、解密、推理、补传和审计闭环。

## 2. 产品重心

Vistral 当前真正要打透的是：

1. 模型主权  
平台掌握成果模型审批、签名、发布、回滚和商业分发。

2. 数据主权  
客户资产默认不出域，训练与验证资产受租户和作业授权控制。

3. 供应商受控协作  
供应商提交初始算法、参与微调和候选模型迭代，但不拥有生产发布权。

4. 边缘安全交付  
模型包加密、签名、授权设备拉取、边缘验签、断网补传、审计追溯形成闭环。

## 3. 核心能力四支柱

### 3.1 主权控制

- 平台掌握代码、数据、模型、密钥和收费权
- 模型包统一由平台验签、签名、发布
- 发布范围可限制到租户、设备、版本

### 3.2 受控协作

- 平台 / 供应商 / 买家三方分工明确
- 训练 worker 只按作业授权拉取训练资产和基线模型
- 候选模型自动回收入库并进入审批流

### 3.3 边缘交付

- 模型加密下发到授权设备
- 边缘侧完成验签、解密、推理与结果回传
- 支持断网补传和长期运行审计

### 3.4 审计与治理

- 关键动作都有审计事件
- 结果、模型、任务、发布动作可追溯
- 适合铁路/政企内网/监管要求较强的交付环境

## 4. 角色价值主张

### 4.1 平台方（platform_admin / platform_operator / platform_auditor）

平台方得到的是：

- 对数据、模型、密钥、发布和收费的最终控制权
- 一个可审计、可回滚、可授权的模型交付控制面
- 对供应商协作过程、成果交付边界和商业分发节奏的持续控制

平台方首条成功路径：

1. 查看待审模型和候选训练作业
2. 结合验证资产与结果做审批判断
3. 将模型或流水线发布到授权租户/设备
4. 在审计与设备页验证发布范围和执行状态

### 4.2 供应商（supplier_demo / supplier_engineer）

供应商得到的是：

- 一个受控但清晰的算法协作入口
- 明确的模型提交、训练协作、候选回收和审批状态反馈
- 不需要接触生产发布与密钥体系，也能持续交付算法价值

供应商首条成功路径：

1. 提交初始算法或预训练模型包
2. 查看训练作业、候选模型和审批进度
3. 按平台要求继续调参与补充说明

### 4.3 买家 / 客户用户（buyer_operator / buyer_auditor）

买家得到的是：

- 数据不出域前提下的资产上传、任务执行和结果查看能力
- 明确知道自己的数据将用于训练、验证还是推理
- 明确知道哪些模型已授权、哪些设备可执行

买家首条成功路径：

1. 上传图片/视频或 ZIP 数据集包
2. 选择推理任务或为训练准备资产
3. 查看结果与截图摘要
4. 追踪设备和授权状态

### 4.4 授权边缘设备

边缘设备得到的是：

- 明确的授权边界和模型拉取方式
- 可离线执行、可补传、可追溯的运行链路
- 平台密钥体系和生产管理入口始终由平台控制

## 5. 产品设计必须遵守的落地原则

### 5.1 首页必须先告诉用户“你是谁、你该做什么”

首页首先回答：

- 当前登录角色是谁
- 该角色在平台中的权责边界
- 第一条推荐路径是什么
- 当前最值得点击的 1-3 个动作是什么

### 5.2 文案必须用业务与治理语言，不用泛化平台语言

优先用这些表达：

- 审批发布
- 授权设备
- 受控协作
- 数据不出域
- 候选模型
- 审计闭环

避免首页和导航过度使用这些泛化表达：

- workspace
- project
- experiment
- generic deploy
- playground

### 5.3 每个角色只暴露最短路径，不暴露全部系统能力

- 买家优先看到资产、任务、结果
- 供应商优先看到模型、训练协作、流水线
- 平台优先看到审批、训练控制面、发布、审计

### 5.4 每个关键对象都要解释“来源、范围、后果”

必须让用户一眼知道：

- 这个资产是训练/验证/推理用途
- 这个模型是初始算法/候选/已发布
- 这个任务跑在谁的设备上
- 这个发布面向哪些租户和设备

### 5.5 资产与发布都要有“先验证再交付”的默认心智

- 资产上传后要立即告诉用户这条记录的用途、资源数、下一步入口
- 训练 / 验证资产优先支持 ZIP 数据集包，方便用户一次性提交多层目录和多资源样本
- 流水线和模型发布前先通过任务执行做一次验证，再进入正式发布动作
- 发布动作默认强调“发布到哪些租户 / 哪些设备”，而不是只强调版本号

## 6. 首屏与导航优化要求

### 6.1 登录页

- 不只展示账号密码，还要展示每个账号对应的角色价值
- 首句文案聚焦“主权 + 受控交付”

### 6.2 工作台

- 角色化 Hero：说明当前角色的职责、收益、下一步
- 角色化快速动作：每个角色只给最相关的 3-4 个入口
- 统一价值卡：反复强化模型主权、数据主权、受控协作、边缘交付

### 6.3 空状态

- 不能只写“暂无数据”
- 必须明确“为什么为空”和“下一步去哪”

## 7. 开发优先级（由定位反推）

优先做：

1. 角色化首页与空状态
2. 模型审批 / 候选 / 发布链路可视化
3. 训练作业输入来源、授权范围、输出摘要可解释
4. 设备授权与边缘运行状态表达

## 8. 对外介绍统一说法

推荐口径：

> Vistral 面向铁路和政企内网环境，重点解决数据不出域、供应商受控协作、模型加密交付、边缘审计追溯和平台侧成果控权问题。
