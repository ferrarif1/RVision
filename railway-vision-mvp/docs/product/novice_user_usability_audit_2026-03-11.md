# 新用户可用性审查（2026-03-11）

- Owner: Engineering
- Status: Draft / In Use
- Last Updated: 2026-03-11
- Scope: 从“只懂少量概念的普通用户”视角，记录当前系统哪里不好用、哪里不明白、哪里会犹豫，以及后续的优化方向
- Persona:
  - 只懂“上传图片、识别车号、训练模型、验证效果”这些少量概念
  - 不理解 `asset_id / model_id / pipeline / dataset_version / worker / readiness` 等术语
  - 希望系统直接告诉我“现在该做什么”，而不是让我自己拼流程
- Non-goals:
  - 不评价底层架构好坏
  - 不评价训练算法效果本身
  - 本文只看“能不能看懂、会不会误用、会不会被流程劝退”

## 1. 总体判断

当前系统已经比早期版本清楚很多，尤其是任务页、结果页、训练页、模型页都开始有“工作台”感觉了。但如果我是一个只知道少量概念的用户，我仍然会在以下时刻卡住：

1. 我能看懂页面在“做什么”，但还不总能看懂“为什么现在要我来这里”。
2. 我能完成主要操作，但很多地方仍需要先理解平台术语，才能放心点击。
3. 我能跟着一个主流程走，但跨页面时仍容易丢失上下文，不确定自己是不是走对了。

换句话说：现在系统已经不像 demo 拼接页，但还没有完全达到“第一次用就自然明白”的程度。

## 2. 最容易卡住的新用户问题

### A. 术语仍然偏平台内部

最容易让新用户愣住的词：

- `asset_id`
- `dataset_version`
- `target_model_code`
- `worker`
- `pipeline`
- `validation_report / readiness`
- `router / expert`

问题不是这些词不能存在，而是它们还会出现在默认操作视图里。对新用户来说，这会产生两个后果：

- 我不确定这些是不是必须懂
- 我不确定填错会发生什么

建议：

- 默认只说“资产 / 数据集版本 / 训练机 / 流水线 / 模型能力”
- 只有进 `技术详情` 时，再露出 `asset_id / model_id / pipeline_id / hash`

## 3. 关键流程里的困惑点

### 3.1 上传资产 -> 创建任务

新用户视角困惑：

- 我已经上传了图片，为什么还要理解“快速识别”和“精确创建任务”两套入口？
- 我不知道什么时候该直接选模型，什么时候该走自动推荐。
- 我看到很多模型卡片，但不确定“选哪版”的依据是什么。

当前已有改进：

- 明确输入 `车号` 后，会直接出现模型选项
- 已支持把上方快速识别模型带入下方精确任务

仍然不够的点：

- 这两个入口虽然已经联动，但从新用户眼里仍然像两套系统
- “快速识别”和“创建任务”之间缺一个更明确的选择说明

建议：

- 合并成一个主入口：
  - `先快速识别`
  - `改用精确任务`
- 默认先展示一个推荐路径，而不是两个并列大块

### 3.2 结果页 -> 下一步

新用户视角困惑：

- 我看到结果了，但不确定这是“能直接用”还是“还要复核”
- 我不知道“导出并送训练中心”和“结果回灌”的关系
- 我不确定现在是应该去训练，还是回去重跑验证

当前已有改进：

- 结果页已经开始用 `车号结果 / 稳定文本 / 待复核 / 平均执行耗时`
- 有 `下一步动作` 工作台

仍然不够的点：

- “下一步动作”和“结果回灌工作台”之间，仍需要用户自己理解关系
- 对新用户来说，“回灌训练”是抽象的

建议：

- 改成两个非常明确的主动作：
  - `结果可信，送去训练`
  - `结果不稳，继续复核`
- “结果回灌工作台”保留，但默认弱化成次级面板

### 3.3 车号文本复核 -> 训练

新用户视角困惑：

- `导出训练资产并打开训练页`
- `导出训练资产并直接创建训练作业`

这两个按钮现在已经比以前清楚，但新用户第一次看仍会想：

- 两个按钮到底差在哪里？
- 如果我只是想“继续训练”，为什么要先理解“训练资产”？

建议：

- 用更接近动作结果的命名：
  - `先准备训练数据`
  - `现在开始训练`

### 3.4 训练 -> 验证 -> 审批 -> 发布

这是当前最容易让新用户感觉“专业，但复杂”的一段。

新用户视角会问：

- 训练成功后，为什么还要验证？
- 验证通过后，为什么还要审批？
- 发布工作台和审批工作台有什么区别？

当前已有改进：

- 审批工作台、发布工作台都已经做成了更明确的工作台
- 训练页里也有候选模型验证入口

仍然不够的点：

- “训练成功 -> 候选模型 -> 审批 -> 发布”的概念层次，仍然需要用户自己理解
- 还缺一个全局统一的“模型生命周期说明”

建议：

- 在训练成功后的下一步卡里，固定解释：
  - `训练成功 = 产出候选模型`
  - `验证 = 看这版模型在真实样本上是否可用`
  - `审批 = 平台确认它值得进入正式交付`
  - `发布 = 把它授权给设备或买家正式使用`

## 4. 最像“大厂产品还没做完”的细节

### 4.1 页面太多，但跨页意图还不够强

虽然现在大部分页面都已经拆成工作区，但新用户仍要自己记：

- 任务在任务页
- 结果在结果页
- 训练在训练页
- 审批在模型页
- 发布在模型页或流水线页

建议：

- 每个关键动作都明确写出“你完成这个动作后会去哪里”
- 页面之间形成固定的下一页跳转，不让用户自己回忆

### 4.2 默认仍有一些“像控制台，不像产品”的信息

典型表现：

- 某些 `badge`、计数和状态还偏系统内部视角
- `技术详情` 虽然已经后置，但仍有少量默认摘要太工程化

建议：

- 所有默认摘要都回答三个问题：
  - 我现在在哪一步？
  - 这一步结果怎么样？
  - 我下一步该做什么？

### 4.3 页面结构清楚了，但心智地图还不够强

新用户未必知道：

- 为什么“模型中心”和“流水线中心”要分开
- 为什么“训练中心”和“车号文本复核”不是一页

建议：

- 在这些页面 Hero 下方加一句“这个页面只负责什么，不负责什么”

## 5. 当前最优先的可用性问题排序

### P0. 继续减少用户必须理解的术语

Done When:

- 默认视图里进一步减少 `asset_id / model_id / pipeline_id / dataset_version / worker`
- 页面文案统一使用更自然的业务语言

### P0. 让主流程只剩一条默认推荐路径

Done When:

- 上传资产后，系统更明确地推荐“直接识别 / 去训练 / 去验证”的单一路径
- 减少新用户一上来就做“模式选择”

### P1. 把训练后治理链路讲明白

Done When:

- 训练成功 -> 验证 -> 审批 -> 发布 的差别，在 UI 中被解释清楚
- 不需要用户靠文档才能理解

### P1. 把“回灌训练”说成人能懂的话

Done When:

- 新用户看到“把确认结果变成训练数据”时，能立刻理解这是把已经确认的结果继续用于下一轮训练

2026-03-11 状态补记：

- 第一轮术语减负已开始落地。
- 复核页主按钮已改成“先准备训练数据 / 现在开始训练”。
- 结果页已把“结果回灌工作台”改成“把确认结果变成训练数据”。
- 训练页里的“候选模型”在高频入口上开始改叫“待验证模型 / 新模型”。

## 6. 推荐的后续优化顺序

1. 先继续压默认术语，把高频页面再去一轮技术化。
2. 再把“上传资产 -> 识别 -> 复核 -> 训练 -> 验证 -> 审批 -> 发布”做成更明显的单一路径引导。
3. 然后补“生命周期解释文案”，让模型从候选到正式发布的逻辑一眼能看懂。
4. 最后再做截图式 onboarding，让新用户不需要读长文档也能上手。

## 7. 这份审查如何使用

- 进入下一轮迭代时，优先拿这份文档里的 `P0 / P1` 项落任务。
- 每次完成一轮页面收边，都回头更新这里的“仍然困惑”项。
- 如果用户提出新的“哪里看不懂”，优先补进本文件，而不是只留在对话里。
